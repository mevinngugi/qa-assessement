ID: Bug-VL-QA-009
Title: User with an extra lengthy password does not receive a verification email and account is not created
Severity: High 
Area: Sign-up 
Steps to Reproduce:
1. Open the Valr homepage in a web browser.
2. Look for the sign-up button on the homepage.
3. Click on the sign-up button.
4. User is redirected to the sign-up page.
5. Fill in the required fields with valid data but for the password field, fill in an extra lengthy password (256 characters long).
6. Click on the "Next" button to proceed to the next step of the sign-up process.
Expected Result:
1. The "Next" button is disabled.
2. The user is shown an error message relating to a max length requirement for the password field
Actual Result:
1. The "Next" button is enabled.
2. The user is not shown any error message.
3. The user is presented with the screen showing that they have been sent an email to verify their email address, but the email is never received. The account was not created because a forgot password attempt with the same email did not receive a reset email.
Risk/Impact: High - This is a critical issue as it prevents users from creating an account with long and secure password of their choice and accessing the platform, which can lead to loss of potential customers and damage to the company's reputation.

ID: Bug-Explanatory-001
Title: When a user tries to access an invalid trading pair url, they are not shown a 404 page but instead are shown a page with a header with content. Just the loading bubble.
Severity: low 
Area: Navigation 
Steps to Reproduce:
1. Open the Valr homepage in a web browser.
2. In the url, try to access an invalid trading pair url - https://www.valr.com/en/exchange/KES/ETH
Expected Result:
1. The user is shown a 404 page indicating that the page they are trying to access does not exist.
Actual Result:
1. The user is shown a page with a header with content. Just the loading bubble.
Risk/Impact: Low - This is a minor issue as it does not prevent users from accessing the platform but may cause confusion as the user might be waiting for the page to load and not understand that the page they are trying to access does not exist.

ID: Bug-API-001
Title: The documentation on Order book states that "Returns a list of the top 40 bids and asks in the order book." but the API returns 41 bids and 41 asks.
Severity: high 
Area: Order book API endpoint 
Steps to Reproduce:
1. Make a get request to the order book endpoint - https://api.valr.com/v1/public/:currencyPair/orderbook
2. Observe the number of bids and asks returned in the response.

Expected Result:
1. Expect the API to return a list of the top 40 bids and asks in the order book as stated in the documentation.    
Actual Result:
1. The API returns 41 bids and 41 asks.
Risk/Impact: High - This is a critical issue as it violates the documented behavior and may cause unexpected results for users relying on the API.
